# MySQL

---

## 1. 索引结构

### B+Tree 特点

- 非叶子节点只存键值（不存数据），单个节点可容纳更多键 → **树更矮**
- 叶子节点形成有序**双向链表** → 范围查询只需找到起点然后遍历
- 叶子节点存完整行数据（聚簇）或主键 ID（二级索引）

### 为什么不用其他树

| 结构 | 问题 |
|------|------|
| 二叉搜索树 | 退化为链表 O(n) |
| AVL 树 / 红黑树 | 每个节点两子节点 → 树太高 → IO 次数多 |
| B 树 | 非叶也存数据 → 高度更高、范围查询需中序遍历回溯 |
| Hash | 无法范围查询、不支持排序、不能部分匹配 |

---

## 2. 聚簇索引 vs 二级索引

- **聚簇索引**：叶子节点存储**完整行数据**（InnoDB 主键索引），每表只能有一个
- **二级索引**：叶子节点存储**主键 ID**，查询非索引列需**回表**查聚簇索引
- **覆盖索引**：查询列全在二级索引内 → 不回表（Using index）

### 为什么建议用自增主键

- 插入时总是在最后追加，减少页分裂和索引重组
- UUID/随机值 → 随机插入 → 频繁页分裂 → 索引碎片 → 性能下降

## 3. 最左前缀原则

联合索引 `(a, b, c)`：

```
WHERE a = 1                      ✅ 用到 a
WHERE a = 1 AND b = 2            ✅ 用到 a, b
WHERE a = 1 AND b = 2 AND c = 3  ✅ 用到 a, b, c
WHERE b = 2                      ❌ 最左列 a 缺失
WHERE a = 1 AND c = 3            ✅ 用到 a（b 缺失，c 无法使用）
WHERE a = 1 AND b > 2 AND c = 3  ✅ 用到 a, b（断在范围查询，c 无法使用）
```

## 4. 索引失效场景

| 场景 | 原因 |
|------|------|
| LIKE `'%abc'` 或 `'%abc%'` | 最左模糊不满足最左匹配 |
| 索引列上做计算 `WHERE age+1=20` | 不是原始索引值 |
| 索引列上使用函数 `WHERE YEAR(date)=2024` | 同上 |
| 隐式类型转换 `WHERE phone=13800138000` | varchar 列被转为 int |
| OR 两侧有非索引列 | 需全表扫描非索引条件 |
| `!=` / `<>` / `NOT IN` | 除覆盖索引外一般不走索引 |
| `IS NULL` / `IS NOT NULL` | 视数据分布可能不走 |
| 联合索引不满足最左前缀 | 见上节 |

## 5. EXPLAIN 关键字段

| 字段 | 含义 | 重点关注 |
|------|------|----------|
| type | 访问类型 | system > const > eq_ref > ref > range > index > **ALL** |
| key | 实际使用的索引 | 是否命中预期索引 |
| rows | 估计扫描行数 | 越小越好 |
| Extra | 额外信息 | Using index(覆盖索引) / Using filesort(文件排序) / Using temporary(临时表) |

## 6. 事务 ACID

| 特性 | 说明 | 实现机制 |
|------|------|----------|
| 原子性 (A) | 要么全成功，要么全回滚 | undo log |
| 一致性 (C) | 事务前后数据满足所有约束 | 由其他三个特性保证 |
| 隔离性 (I) | 并发事务间不可见 | MVCC + 锁 |
| 持久性 (D) | 提交后永久保存 | redo log |

---

## 7. 隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| Read Uncommitted | ✅ | ✅ | ✅ |
| Read Committed | ❌ | ✅ | ✅ |
| Repeatable Read (InnoDB 默认) | ❌ | ❌ | 部分解决 |
| Serializable | ❌ | ❌ | ❌ |

### MVCC 原理

- 每行数据有隐藏列：`DB_TRX_ID`（最近修改事务 ID）、`DB_ROLL_PTR`（undo log 回滚指针）
- ReadView 包含当前活跃事务 ID 列表，通过比对可见性规则判断快照

### Read Committed vs Repeatable Read（MVCC 角度）

- RC：每次 SELECT 生成新的 ReadView → 读到其他事务已提交的更新 → 不可重复读
- RR：事务开始时生成 ReadView，整个事务用同一个 → 可重复读

## 8. redo log / undo log / binlog

| | redo log | undo log | binlog |
|--|----------|----------|--------|
| 层级 | InnoDB 引擎层 | InnoDB 引擎层 | MySQL Server 层 |
| 内容 | 物理日志：页的修改 | 逻辑日志：修改前的数据 | 逻辑日志：SQL 语句 |
| 作用 | 崩溃恢复（持久性） | 回滚 + MVCC | 主从复制、数据恢复 |
| 写入方式 | 循环写（固定大小） | — | 追加写 |

## 9. SQL 优化思路

```
慢查询日志 → EXPLAIN 分析 → 
  type=ALL/index → 加索引
  索引失效 → 改写 SQL
  Using filesort → 优化 ORDER BY
  Using temporary → 优化 GROUP BY/DISTINCT
  回表次数多 → 覆盖索引
  大偏移量分页 → 基于游标的翻页
```

## 10. 分库分表

### 方案

- **垂直分库**：按业务拆（订单库、用户库）
- **垂直分表**：大字段或冷热数据分离
- **水平分库分表**：按某字段（user_id）哈希取模路由

### 带来的问题

| 问题 | 解决方案 |
|------|---------|
| 分布式 ID | 雪花算法（Snowflake） |
| 跨库 JOIN | 应用层组装 / ES 冗余 |
| 分布式事务 | Seata（AT / TCC / SAGA） |
| 跨节点分页/排序 | 数据量少用中间件聚合，量大用 ES |
| 扩容 | 一致性哈希 / 倍数扩容减少数据迁移 |
