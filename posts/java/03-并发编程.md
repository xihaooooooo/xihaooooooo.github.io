## 1. 线程生命周期

### 1.1 六种状态

Java 线程在 `Thread.State` 枚举中定义了六种状态：

```
NEW ──start()──▶ RUNNABLE ──等待锁──▶ BLOCKED ──获得锁──▶ RUNNABLE
                   │                                      │
                   ├── wait()/join()/park() ──▶ WAITING ──notify()/unpark()──▶
                   │                                      │
                   ├── sleep(ms)/wait(ms) ──▶ TIMED_WAITING ──超时/唤醒──▶
                   │                                                          │
                   └── run()结束 ──▶ TERMINATED
```

- **NEW**：`new Thread()` 创建但尚未 `start()`。线程对象已存在，但底层 OS 线程还未创建。
- **RUNNABLE**：调用 `start()` 后进入此状态。注意：Java 的 RUNNABLE 合并了 OS 的 Ready（就绪等待 CPU）和 Running（正在执行），因为 Java 层面无法精确区分。
- **BLOCKED**：线程试图获取 synchronized 锁但锁被其他线程持有。**注意**：只有 synchronized 会导致 BLOCKED，`Lock.lock()` 等 JUC 锁走的是 WAITING（内部用 `LockSupport.park()`）。
- **WAITING**：无限期等待，直到被其他线程显式唤醒。进入方式：`Object.wait()`、`Thread.join()`、`LockSupport.park()`。退出：`notify()`/`notifyAll()`、`unpark()`、join 的线程终止。
- **TIMED_WAITING**：带超时的等待。`Thread.sleep(ms)`、`Object.wait(ms)`、`Thread.join(ms)`、`LockSupport.parkNanos()`。超时自动回到 RUNNABLE，不依赖外部唤醒。
- **TERMINATED**：线程执行完 `run()` 或异常退出。terminated 的线程无法再次 `start()`，会抛 `IllegalThreadStateException`。

### 1.2 常见误区

**sleep vs wait 的核心区别**：
- `sleep()` 是 Thread 的静态方法，**不释放锁**，抱着锁睡觉
- `wait()` 是 Object 的方法，**释放锁**，必须在 synchronized 块中调用，否则抛 `IllegalMonitorStateException`
- `sleep` 醒来后回到 RUNNABLE；`wait` 被 notify 后先进入 BLOCKED（需要重新竞争锁），拿到锁后才回到 RUNNABLE

**BLOCKED vs WAITING 的区别**：
- BLOCKED 是"等在 synchronized 门外"，拿不到锁就堵在这里，拿到锁立即进入
- WAITING 是"等着别人来叫我"，主动调了 wait/join/park，必须等 notify/unpark

**notify vs notifyAll**：
- `notify()` 随机唤醒一个等待线程（不保证公平，由 JVM 决定）
- `notifyAll()` 唤醒所有等待线程，它们重新竞争锁
- 实践中除了确定只唤醒一个的场景，**推荐用 notifyAll**，避免"信号丢失"导致线程永久阻塞

## 2. synchronized 锁升级（JDK 1.6+）

### 2.1 Mark Word：锁状态的存储工

每个 Java 对象的对象头中有一个 **Mark Word**（32 位 JVM 中 32 bit，64 位中 64 bit），JVM 在这里用**最低几位标记**来记录锁状态：

| 锁状态 | 标志位 | Mark Word 存储内容 |
|--------|--------|-------------------|
| 无锁 | 01 | 31 位 hashcode（如果有）、分代年龄 |
| 偏向锁 | 01 | 线程 ID（54 位）+ epoch + 分代年龄 + 偏向标记 1 |
| 轻量级锁 | 00 | 指向线程栈中 Lock Record 的指针 |
| 重量级锁 | 10 | 指向堆中 Monitor 对象的指针 |
| GC 标记 | 11 | 空（GC 期间标记） |

关键：无锁和偏向锁的标志位都是 01，区别在于"是否设置了偏向标记位"。这就是对象头能承载锁升级的物质基础——几比特标志位，语义全变。

### 2.2 偏向锁：无竞争时的极致优化

**适用场景**：只有一个线程反复获取同一把锁（最常见——大部分同步块事实上只有一个线程在访问）。

**加锁过程**：
1. 检查 Mark Word 的偏向标记是否为 1，且线程 ID 指向自己 → 已持有，无需同步
2. 如果线程 ID 为空 → CAS 将自己的线程 ID 写入 Mark Word → 成功即获得偏向锁
3. 如果线程 ID 是其他线程 → 发生**偏向撤销**（revocation），升级为轻量级锁

**撤销过程**（需要在 SafePoint 进行）：
- JVM 暂停持有偏向锁的线程（即使是活跃线程也要停下）
- 检查原线程是否还存活、是否还在同步块内
  - 已退出 → 恢复无锁状态
  - 还在持有 → 升级为轻量级锁
- SafePoint 的代价不小，这也是为什么**偏向锁在激烈竞争场景下反而拖慢性能**

**JDK 15 起偏向锁默认禁用**：现代应用（尤其是线程池、异步并发）中"一把锁只有一个线程用"的场景越来越少，偏向锁的撤销成本高于收益。通过 `-XX:+UseBiasedLocking` 可重新启用。

### 2.3 轻量级锁：CAS 自旋

**适用场景**：多个线程竞争但**交替执行**（不是同时抢），锁持有时间短。

**加锁过程**：
1. 当前线程在栈帧中创建 **Lock Record**，拷贝 Mark Word 到 Lock Record 中（称为 Displaced Mark Word）
2. CAS 尝试将对象头的 Mark Word 替换为**指向 Lock Record 的指针**
3. CAS 成功 → 线程获得轻量级锁，Mark Word 标志位变为 00
4. CAS 失败 → 存在竞争，进入**自旋**
   - 先自适应自旋（Adaptive Spinning）：JVM 根据历史自旋成功率决定自旋次数，不是固定值
   - 自旋期间锁被释放 → CAS 成功
   - 自旋失败次数达到阈值 → 膨胀为重量级锁

**解锁过程**：CAS 将 Displaced Mark Word 写回对象头。如果失败（说明已膨胀为重量级锁），走重量级解锁流程。

轻量级锁的本质是"**拿 CPU 换阻塞**"——自旋消耗 CPU，但避免了线程切换的成本（上下文切换约 1-5 微秒）。所以轻量级锁适合锁持有时间短的场景，如果锁持有时间长，自旋就是纯浪费 CPU。

### 2.4 重量级锁：操作系统互斥量

膨胀为重量级锁后，Mark Word 存储指向 **ObjectMonitor** 的指针。ObjectMonitor 包含：
- **\_owner**：持有锁的线程
- **\_EntryList**：BLOCKED 状态的线程队列（竞争者）
- **\_WaitSet**：调了 `wait()` 的线程集合（WAITING 状态）

竞争失败的线程被**挂起（park）**，不再自旋。锁释放时从 \_EntryList 中挑选一个线程唤醒。线程切换涉及用户态→内核态、上下文保存恢复，代价高。

**锁膨胀条件**：自旋失败超过阈值、或检测到自旋有意义（owner 线程被调度走了），JVM 主动将轻量级锁膨胀为重量级锁。

### 2.5 锁降级？

官方说法是"单向不可降级"，但有一个例外：**发生 SafePoint 时，如果重量级锁的所有线程都已退出，且对象未膨胀到严重的竞争程度，JVM 可能将 Monitor 回收，对象头恢复到无锁状态**。例如 HotSpot 在 GC 的偏向锁批量撤销（Bulk Revocation）中会重置偏向状态。但这不是常规流程，面试中答"不可降级"即可。

### 2.6 synchronized 的三个使用位置

```java
// 1. 普通方法 — 锁当前实例
public synchronized void method() { }

// 2. 静态方法 — 锁 Class 对象
public static synchronized void method() { }

// 3. 同步块 — 锁指定对象
synchronized (obj) { }
```

底层编译后，方法级的 synchronized 由方法标志 `ACC_SYNCHRONIZED` 标记，JVM 执行时隐式加锁/解锁；同步块则编译为 `monitorenter` 和 `monitorexit` 两条字节码指令（编译器会插入两条 monitorexit——正常出口和异常出口，保证锁一定被释放）。

### 2.7 JIT 的锁优化

除了锁升级，JIT 即时编译还会做：

- **锁消除**：通过逃逸分析，发现锁对象只在单线程内使用，直接去掉同步（如 `Vector` 在局部变量中使用，可以消除内部 synchronized）
- **锁粗化**：连续的加锁/解锁合并为一个大锁，减少获取释放次数（如循环体内反复 append 到 StringBuffer）

这些优化完全透明，不需要改任何代码。

## 3. volatile

### 3.1 可见性：为什么普通变量不可见

先看一个经典问题：

```java
public class VisibilityDemo {
    private static boolean flag = false;  // 普通变量

    public static void main(String[] args) {
        new Thread(() -> {
            while (!flag) {}  // 线程 B 可能永远看不到 flag 变 true
        }).start();

        Thread.sleep(1000);
        flag = true;  // 线程 A 修改
    }
}
```

线程 B 可能永远不会退出循环。原因：JMM（Java Memory Model）中，每个线程有自己的**工作内存（CPU 缓存/寄存器）**，普通变量修改后线程 B 不一定立即看到。

```
线程 A                    主内存                线程 B
  │                        │                      │
  │  flag = true ──────▶  flag = true             │
  │  (可能留在 A 的                         flag = false
  │   工作内存中)                            (B 缓存中的旧值)
```

**volatile 的解决方案**：

```java
private static volatile boolean flag = false;
```

- **写** volatile 变量时，JVM 插入 **store 内存屏障**，强制将工作内存数据刷回主内存
- **读** volatile 变量时，JVM 插入 **load 内存屏障**，强制从主内存重新读取，使线程工作内存中的缓存值失效

效果：线程 A 写入 volatile 变量后，线程 B 再读这个变量，一定能看到最新值。

### 3.2 禁止指令重排：DCL 为什么需要 volatile

JVM 为了优化，可能在不影响单线程结果的前提下**重排指令**。看 DCL 单例：

```java
public class Singleton {
    private static volatile Singleton instance;  // 必须 volatile！

    public static Singleton getInstance() {
        if (instance == null) {                     // ① 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {             // ② 第二次检查
                    instance = new Singleton();     // ③ 这里可能会重排
                }
            }
        }
        return instance;
    }
}
```

`instance = new Singleton()` 在字节码层面是三步：

```
1. 分配内存空间
2. 初始化对象（执行构造方法）
3. 将引用指向内存地址
```

但 JIT 编译器可能**重排为 1→3→2**：先分配内存、赋引用，再初始化。此时如果线程 B 执行到 ①，看到 instance 不为 null（步骤 3 已执行），但对象还没初始化完（步骤 2 未执行），线程 B 拿到一个**半成品的对象**，调用其方法就崩了。

**volatile 禁止了这种重排**：在 volatile 写前后插入内存屏障，保证 1→2→3 的顺序不会被颠倒。

```java
// 等价于
memory_barrier();          // StoreStore 屏障
instance = new Singleton();
memory_barrier();          // StoreLoad 屏障
```

### 3.3 不保证原子性

volatile 解决了可见性和有序性，但**不解决原子性**。典型反例：

```java
volatile int count = 0;

// 100 个线程各执行 1000 次 count++，结果 < 100000
count++;  // 读→改→写 三段，不是原子的
```

`count++` 看似一行代码，实际是三条指令：

```
getfield    count   // 读
iconst_1            // 准备常量 1
iadd                // 加
putfield    count   // 写
```

线程 A 读到 0，还没来得及写，线程 B 也读到 0，两个线程各加 1 后都写回 1 —— 丢了一次递增。

需要原子性时用 `AtomicInteger` 或 `synchronized`：

```java
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();  // CAS 保证原子性
```

### 3.4 使用场景

**适合 volatile 的场景**：

- **状态标志位**：`volatile boolean isRunning`，一个线程改，其他线程读
- **DCL 单例**：防止指令重排导致半成品对象
- **volatile bean 模式**：将一组不可变数据的引用标记为 volatile（一次性写入，多次读取）

**不适合 volatile 的场景**：

- 任何依赖当前值的复合操作：`count++`、`if (flag) flag = false`
- 多个 volatile 变量之间有依赖关系

**happens-before 规则**：对一个 volatile 变量的写操作 happens-before 后续对该变量的读操作。这是 JMM 规范保证的，也是上面所有讨论的理论基础。

## 4. CAS（Compare And Swap）

### 4.1 什么是 CAS

CAS 是一条 CPU 原子指令，全称 **Compare And Swap**（比较并交换）。它的语义是：

> 如果内存位置 V 的值等于期望值 A，就把它修改为 B；否则什么都不做。**无论成败，都返回 V 的旧值。**

```
CAS(V, A, B) → 如果 *V == A，则 *V = B，返回旧值
```

整个"比较→交换"过程由 **CPU 硬件保证原子性**，不会被线程调度打断。Java 的 CAS 最终依赖一条 CPU 指令 —— `lock cmpxchg`。

### 4.2 从 Java 到 CPU：调用链路

以 `AtomicInteger.incrementAndGet()` 为例，调用链路分三层：

**第一层 — AtomicInteger**：用户调 `incrementAndGet()`，它内部委托给 `Unsafe.getAndAddInt(this, valueOffset, 1)`。`valueOffset` 是 `value` 字段在对象内存中的偏移量，通过 `Unsafe.objectFieldOffset()` 在类加载时计算好，后续所有 CAS 操作都靠这个偏移量定位要修改的内存地址。

**第二层 — sun.misc.Unsafe**：`getAndAddInt` 的核心是一个 do-while 循环 —— 先用 `getIntVolatile` 读到当前值（volatile 语义，保证是最新值），然后调用 `compareAndSwapInt` 尝试把值从旧的改成新的。如果这期间有别的线程改过值，CAS 返回 false，循环继续重试，直到成功。这就是"自旋"的由来。

**第三层 — native → CPU 指令**：`compareAndSwapInt` 是 native 方法，由 JVM 的 C++ 代码直接调用 CPU 的 `lock cmpxchg` 指令。`lock` 前缀做两件事：一是锁定缓存行，确保该地址上的读-改-写不会被打断；二是充当内存屏障，让这次写操作对所有线程立即可见。所以在 x86 上，一条指令就同时解决了原子性和可见性两个问题。

### 4.3 三大核心问题

**① ABA 问题**

线程 A 读取值 X，线程 B 将 X 改为 Y 再改回 X，此时线程 A 执行 CAS 仍然成功 —— 但数据在此期间已经被人动过了。"狸猫换太子"。

对基本类型（int、long）通常无害，因为值相同就意味着状态相同。但对引用类型可能致命。举个例子：无锁栈的节点是 `node1 → node2 → node3`，线程 A 读到栈顶 `node1`，正准备 CAS 将栈顶改为 `node2`；与此同时线程 B 连续弹出 `node1` 和 `node2`，又 push 回 `node1`。此时栈顶又变成了 `node1 → node3`，但 `node2` 已经被回收释放了。线程 A 的 CAS 此时仍然成功（值还是 `node1`），但 `node1.next` 指向的是已被释放的 `node2`，变成了野指针。

**解决**：带版本号的 CAS，典型实现是 `AtomicStampedReference`，每次更新同时递增一个 stamp 版本号，CAS 时同时比较引用和版本号，ABA 问题被彻底消除。还有简化版的 `AtomicMarkableReference`，只用一个 bool 标记"有没有被改过"。

**② 自旋开销**

CAS 失败后循环重试（自旋），如果竞争激烈，多个线程长时间自旋会大量消耗 CPU —— 大量线程在空转，CPU 飙高但没干正事。

解决方向有三种：第一，JVM 内置了**自适应自旋**，根据上次自旋是成功还是失败来动态调整本次自旋次数；第二，业务代码可以加**退避策略**，失败多次后主动 `yield()` 让出 CPU，甚至短暂 sleep；第三，高竞争场景下直接**改用锁**，让竞争失败的线程阻塞挂起而不是空转。

**③ 只能保证一个共享变量的原子操作**

CAS 操作的是单个内存地址。如果要原子地修改多个变量，CAS 无能为力。解决方案就两种：要么用 `AtomicReference` 把多个变量封装成一个对象，CAS 更新整个对象的引用；要么直接用锁。

### 4.4 AtomicInteger 设计要点

`AtomicInteger` 的核心组成非常简单：一个 `volatile int value` 存实际值，一个 `Unsafe` 实例用来操作内存，一个 `valueOffset` 记录 value 字段在对象中的内存偏移。所有方法本质上就是"读 value → CAS 尝试更新 → 失败重读重试"这个循环。

几个值得关注的设计细节：

- **value 是 volatile 的**：保证每次读到最新值，CAS 失败后立即能看到新值开始下一轮重试，而不是读到过期的缓存值反复重试
- **lazySet（JDK 9+）**：普通 volatile 写会触发完整的 store-load 屏障，开销较大。`lazySet` 用 `putOrderedInt` 实现，只保证 store-store 顺序，不保证对其他线程立即可见，适用于不需要强一致性的"最终写入"场景，性能明显更高
- **valueOffset 用 static 初始化**：通过 `objectFieldOffset` 反射拿到字段偏移，类加载时执行一次，之后所有 CAS 操作都用这个偏移量定位内存地址

### 4.5 CAS 在 JDK 中的应用

- **AtomicInteger / AtomicLong / AtomicBoolean**：无锁的数值和标志位操作，最常见的是计数器、ID 生成器
- **AtomicReference / AtomicReferenceFieldUpdater**：无锁更新对象引用，适合需要原子替换整个对象的场景
- **ConcurrentHashMap**：扩容时用 CAS 设置 `sizeCtl` 字段，确保只有一个线程执行扩容；put 时用 CAS 写入 tab[index] 为 null 的槽位（不加桶锁）
- **AQS（AbstractQueuedSynchronizer）**：ReentrantLock、Semaphore、CountDownLatch 的底层框架，全部靠 `CAS(state)` 来快速抢占同步状态：先 CAS 一次，成功就直接获取锁，失败才进入 CLH 队列排队
- **ConcurrentLinkedQueue**：无锁队列，入队时 CAS 更新 tail 指针，出队时 CAS 更新 head 指针

### 4.6 CAS vs Lock：什么时候用哪个

**CAS（乐观锁，无锁）**：
- 适用于**读多写少**、**竞争不激烈** 的场景
- 不会引起线程阻塞和上下文切换
- 失败后自旋重试，竞争激烈时 CPU 空转
- 无法处理复杂的状态依赖（比如"检查状态 → 修改状态"两步必须原子）

**Lock（悲观锁）**：
- 适用于**写多**、**竞争激烈** 的场景
- 未能获取锁的线程被阻塞（park），不消耗 CPU
- 但阻塞和唤醒涉及上下文切换（用户态 ↔ 内核态），开销较大
- synchronized（偏向锁→轻量锁→重量锁）已经做了三层优化，高竞争下反而更优

**一句话选型**：竞争低用 CAS，竞争高用锁；不确定就先上锁，profile 后发现热点再考虑无锁优化。

## 5. AQS（AbstractQueuedSynchronizer）

### 5.1 设计理念：模板方法模式

AQS 是 JUC（java.util.concurrent）的灵魂框架。ReentrantLock、Semaphore、CountDownLatch、ReentrantReadWriteLock、ThreadPoolExecutor 的 Worker —— 这些类的同步逻辑全部建立在 AQS 之上。

AQS 的核心设计是**模板方法模式**：它把"排队、阻塞、唤醒"这些复杂的并发控制全都做好了（顶层模板），子类只需重写几个简单方法来定义"什么情况下能拿到锁、拿锁后 state 怎么变"（业务逻辑）。具体来说：

- **AQS 负责**：维护同步等待队列（CLH 变体）、线程的阻塞与唤醒、state 的 volatile/CAS 操作、超时和中断处理
- **子类负责**：实现 `tryAcquire`、`tryRelease` 等方法来定义获取/释放的规则

这意味着你写一个自定义同步器只需几十行代码，而不用从零处理 `LockSupport.park()`、队列管理、竞态条件等棘手问题。

### 5.2 三要素详解

**① state（volatile int）—— 同步状态的唯一真相源**

state 是 AQS 中最核心的字段。它的语义完全由子类定义，AQS 只提供三个原子操作方法：

- `getState()`：读 state（volatile 读，每次都从主内存取最新值）
- `setState(newValue)`：写 state（volatile 写，立即可见）
- `compareAndSetState(expect, update)`：CAS 更新 state（只有一个线程能成功地从 expect 改为 update）

同一根 state，语义完全不同：
- ReentrantLock 中：state = 0 表示无锁；state = 1 表示有线程持锁；state > 1 表示持锁线程重入了 state 次
- Semaphore 中：state 表示剩余许可数
- CountDownLatch 中：state 表示还需 countDown 多少次
- ReentrantReadWriteLock 中：state 高 16 位存读锁计数，低 16 位存写锁重入次数

**② CLH 队列 —— 公平高效的等待线**

AQS 维护一个 FIFO 的双向链表（CLH 锁队列的变体），每个节点代表一个等待线程。节点中的 `waitStatus` 字段记录了线程状态（CANCELLED、SIGNAL、CONDITION、PROPAGATE）。

关键细节：队列的 head 总是指向一个特殊的哨兵节点（当前持有锁的线程或已经释放的虚拟节点），真正的等待者从 head 的后继开始。每次释放锁时，AQS 唤醒 head 的后继节点，而不是遍历整个队列，效率很高。

线程入队时，AQS 用 CAS 更新 tail 指针来完成并发安全的链表尾插。出队时，只需移动 head 指针（释放锁的那个线程自己就是 head，直接修改 head 指向后继即可，不存在竞争）。

**③ CAS —— 无锁竞争 state 和队列操作**

获取锁时，AQS 的通用策略是"先 CAS 抢一下，抢不到再排队"：

- CAS 尝试把 state 从 0 改为 1（或从当前值改为 current-1 等）
- 抢到了，当前线程获得锁，无需入队 —— 这是最快的路径
- 没抢到，当前线程被包装成节点，CAS 插入队列尾部，然后 park 挂起

这种"先乐观抢、再悲观等"的设计兼顾了低竞争时的高性能和高竞争时的公平性。

### 5.3 独占模式：ReentrantLock 为例

独占模式下，同一时刻只有一个线程能持有锁。流程：

**获取锁（acquire）**：
1. 调 `tryAcquire(arg)`，由子类实现。以 ReentrantLock 的非公平模式为例：检查 state，如果为 0 就 CAS 设为 1 并设自己为 owner；如果不为 0 但 owner 是自己，就 state+1（可重入）
2. 如果 `tryAcquire` 失败，当前线程包装成 EXCLUSIVE 节点，CAS 入队尾，然后 `acquireQueued` 进入自旋：先检查前驱是不是 head（即排到自己了没），如果是就再 tryAcquire 一次，失败则 park 挂起，等待前驱唤醒
3. 被唤醒后回到步骤 2 的逻辑，重新 check 前驱是不是 head —— 即使被唤醒，也要重新竞争锁（这就是"虚假唤醒"的防御方案）

**释放锁（release）**：
1. 调 `tryRelease(arg)`：state 减 1，如果 state 变为 0 说明完全释放，清除 owner
2. AQS 唤醒 head 的后继节点（调用 `LockSupport.unpark`）
3. 被唤醒的线程从 `park` 中醒来，继续自旋抢锁

**ReentrantLock 公平 vs 非公平的区别**，仅在 `tryAcquire` 的实现：公平版本在 CAS 前多了一步检查"队列里有没有人排在我前面"，有的话直接放弃此次 CAS，入队排到末尾。非公平版本上来就 CAS，不管有谁在排队——所以叫"插队锁"。

### 5.4 共享模式：Semaphore 和 CountDownLatch

共享模式的关键不同：锁（或称"许可"）可以被多个线程同时持有。

**Semaphore**：初始化时设定许可数（permits），这个数字存入 state。`acquire()` 就是 `tryAcquireShared`：检查 state，如果大于 0（还有许可），CAS 减 1，成功则获取许可；如果 state 已经是 0，入队排队。`release()` 就是 CAS 将 state 加 1，然后唤醒队列中等待的线程来竞争这个释放的许可。

**CountDownLatch**：state 初始化为 count 值。`await()` 不是拿锁，而是问"state 是否等于 0"——等于 0 才通过，否则入队挂起。`countDown()` 每次 CAS 将 state 减 1，当 state 减到 0 时，一次性唤醒队列中所有等待线程。注意 CountDownLatch 只能触发一次，state 到 0 后就不能重置了——这就是为什么它能做"发令枪"。

**Semaphore 和 CountDownLatch 的共享释放区别**很关键：Semaphore release 后只唤醒队列中的 head 后继（一个线程拿一个许可）；CountDownLatch 的 countDown 触发 0 后，是**释放链式传播**——队首被唤醒、获取成功、然后队首主动唤醒自己的后继，像多米诺骨牌一样一直传播下去，直到所有等待线程全部被唤醒。

### 5.5 模板方法一览

AQS 定义了完整的获取/释放流程，子类只需重写以下方法中的一个或多个：

**独占模式**（如 ReentrantLock）：
- `tryAcquire(int arg)`：尝试获取锁，返回 true/false
- `tryRelease(int arg)`：尝试释放锁，返回 true/false（state 是否完全释放）
- `isHeldExclusively()`：当前线程是否独占持有

**共享模式**（如 Semaphore）：
- `tryAcquireShared(int arg)`：尝试获取共享许可，返回剩余许可数（负数表示失败）
- `tryReleaseShared(int arg)`：尝试释放共享许可，返回是否允许唤醒后继

AQS 的 acquire 流程本身还拆成了多种变体供上层调用：
- `acquire(arg)`：不可中断的独占获取（中断只打标记，不抛异常，继续自旋）
- `acquireInterruptibly(arg)`：可中断的独占获取（中断就抛 InterruptedException）
- `tryAcquireNanos(arg, nanos)`：带超时的独占获取
- `acquireShared` / `acquireSharedInterruptibly` / `tryAcquireSharedNanos`：对应的共享版本

### 5.6 Condition：条件等待

AQS 中除了 CLH 同步队列，还内置了**条件队列（Condition Queue）**。`Condition` 对象由 `Lock.newCondition()` 创建，每个 Condition 自己维护一条单链表。

当线程调用 `condition.await()` 时：线程先从同步队列中出队（释放锁），然后把自己加入 Condition 的等待队列，park 挂起。当另一个线程调 `condition.signal()` 时：AQS 从 Condition 队列的头部取一个节点，把它**转移到同步队列的尾部**（用 CAS 操作），被转移的线程就能像普通等待锁的线程一样竞争锁了。

注意：被 signal 的线程不是立即执行，而是要重新获取锁——所以 `await()` 返回时，线程已经重新持有了锁，和调用 `await()` 之前的状态一致。

Condition 的最大价值是比 `synchronized + wait/notify` 更灵活：一个 Lock 可以创建多个 Condition（例如 ReentrantLock 的生产者-消费者场景，可以用两个 Condition 分别通知"缓冲区非空"和"缓冲区非满"），而 `synchronized` 只有一个隐式的 wait/notify 条件。

## 6. ReentrantLock vs synchronized

| | synchronized | ReentrantLock |
|--|-------------|---------------|
| 实现 | JVM 层面（monitor） | JDK 层面（AQS） |
| 锁释放 | 自动（代码块结束/异常） | 手动 unlock（finally 中） |
| 可中断 | ❌ | ✅ lockInterruptibly() |
| 可超时 | ❌ | ✅ tryLock(timeout) |
| 公平锁 | ❌ 非公平 | ✅ 可选公平/非公平 |
| 多条件 | ❌ | ✅ 多个 Condition |
| 性能 | 1.6 后近似 | 近似 |

### 逐条展开

**实现层面**：synchronized 是 JVM 原生支持的关键字，锁信息和线程状态记录在对象头的 Mark Word 中，升级过程（偏向锁→轻量级锁→重量级锁）全由 JVM 自动完成，不需要任何代码显式参与。ReentrantLock 是纯 Java 代码实现的，基于 AQS 框架的 state + CLH 队列 + CAS 三要素，在 JDK 层面管理同步状态和等待线程。

**锁释放机制**——这是两者最大的人身安全差异。synchronized 无论正常返回还是异常抛出，JVM 都会自动执行 `monitorexit` 释放锁，不存在忘记释放的可能性。ReentrantLock 必须手动在 finally 块中调 `unlock()`，一旦忘记或代码路径没覆盖到，锁就永远不会释放，其他线程永久阻塞。这个特性在快速 bugfix 和异常路径较多的场景下极易踩坑。综合来看，能用 synchronized 的场景不必强行换 ReentrantLock，"忘加 finally"比"性能差一点"更致命。

**可中断性**：synchronized 中，如果一个线程在等待锁时被调了 `interrupt()`，它不会响应，继续等。只有成功拿到锁的线程在执行同步块时才能感受到中断（自己检查 `interrupted()` 或调用可响应中断的方法）。而 `ReentrantLock.lockInterruptibly()` 允许等待锁的线程响应中断，被 interrupt 时立即抛 `InterruptedException` 从等待中退出。这在需要防止死锁超时、或需要取消后续任务的场景中非常关键。

**可超时**：`ReentrantLock.tryLock(timeout, unit)` 允许指定一个最大等待时间，超时后返回 false，线程可以去做别的或直接报错放弃。synchronized 没有这个能力，要么拿到锁，要么一直等。可超时能让系统在面对下游超时或资源竞争时优雅降级，而不是无限卡死。

**公平性**：synchronized 是非公平的，被唤醒的线程和刚好同时到达的新线程一起抢锁，新线程可能插队。ReentrantLock 支持构造时指定 `fair=true` 开启公平模式，线程严格按等待时间排队。公平锁有代价——每释放一次锁都要从队列头唤醒后继，增加了唤醒和上下文切换频率，吞吐量通常比非公平锁低一次。只有对"顺序公平"有强需求的场景（如支付队列）才需要用公平锁。

**多条件等待**：synchronized 只能配合一个隐式条件队列，通过 `wait()` / `notify()` / `notifyAll()` 操作。这意味着所有等待条件不同的线程都混在同一个等待集中，`notifyAll()` 一唤醒就是全员唤醒，大部分线程检查条件不满足后又继续 wait，产生大量无意义的上下文切换。而 ReentrantLock 可以创建多个 Condition，每个 Condition 对应一种等待条件——比如生产者-消费者场景，可以用 `notEmpty` 和 `notFull` 两个 Condition 分别通知"可以消费了"和"可以生产了"，唤醒精确、效率更高。

**性能对比**：JDK 1.6 之前 synchronized 是纯重量级锁，性能远不如 ReentrantLock。1.6 引入锁升级（偏向锁→轻量级锁→重量级锁）和自适应自旋后，两则性能基本持平。实际上，在低竞争场景下 synchronized 的偏向锁优化甚至略优于 ReentrantLock（避免 AQS 的 CAS 初检开销）；高竞争场景下，ReentrantLock 的灵活 Condition 和超时机制成为选型考量而非纯粹性能问题。

### 选型建议

- **首选 synchronized**：绝大多数场景。自动释放锁、零心智负担、JVM 持续优化
- **用 ReentrantLock** 当且仅当需要以下三者至少之一：可中断拿锁、可超时放弃、多条件精确唤醒
- 公平锁除非有"线程等待顺序必须公平"的硬性需求，否则不建议开

## 7. ThreadLocal

### 7.1 解决的问题

ThreadLocal 为每个线程提供自己独立的变量副本，线程之间互不干扰，从而实现**线程封闭（Thread Confinement）**——不需要加锁，没有竞争，天然线程安全。

常见场景：Web 应用中每个请求对应一个线程，用 ThreadLocal 存当前登录用户信息，整个请求链路中的 DAO -> Service -> Controller 都能访问，不需要层层传参。Spring 的 `RequestContextHolder`、`TransactionSynchronizationManager`（管理数据库连接和事务）就是基于 ThreadLocal 实现的。

### 7.2 内部结构

每个 Thread 对象中持有一个 `ThreadLocalMap`（不是 HashMap，是完全独立实现的散列表）。这个 Map 的 key 是 ThreadLocal 对象本身（通过弱引用包装），value 是线程私有的那个变量副本。

**WeakReference 的设计意图**：如果把 key 设为强引用，那么即使 ThreadLocal 对象在业务代码中已经出了作用域、没有地方再引用它了，只要线程还存活（线程池中线程往往长期存活），ThreadLocalMap 就还持有它的强引用，导致 ThreadLocal 对象本身无法被 GC。弱引用解决的就是这个问题——ThreadLocal 对象在外部没有再被引用时，可以被 GC 回收。

Entry 的继承结构与问题根源：

- Entry 继承自 `WeakReference<ThreadLocal<?>>`，key 字段是弱引用
- 但 value 字段是普通强引用，Entry 对象本身也是 ThreadLocalMap 内部数组的强引用
- 当 key（ThreadLocal）被 GC 回收后，Entry 的 key 变成了 null，但 value 仍然被 Entry → 数组 → Thread → ThreadLocalMap 这条强引用链指着，无法回收

这就是经典的内存泄漏路径：使用 ThreadLocal 的线程长期存活（如线程池核心线程）→ key 被 GC → value 永远留在内存中。

### 7.3 解决：remove() 与惰性清理

**正确用法**：

```java
ThreadLocal<User> currentUser = new ThreadLocal<>();
try {
    currentUser.set(user);
    // 业务逻辑...
} finally {
    currentUser.remove();  // 必须有！
}
```

**惰性探测清理**：即使忘记 remove()，ThreadLocal 也不是完全没有自救机制。在每次 `get()` 或 `set()` 时，ThreadLocal 使用开放地址法查找 key 的过程中，如果沿途发现 key 为 null 的 Entry（即"脏 Entry"），会顺便清理掉。此外，当哈希冲突严重触发扩容时，也会统一清理所有脏 Entry。

但惰性清理不可靠——如果线程处理完一个任务后长时间没有后续的 get/set，脏 Entry 就一直占着内存，而线程池的线程可以活几小时甚至几天，大量脏数据堆积会导致 OOM。结论：**remove() 必须显式调用**，不能依赖惰性清理。

### 7.4 InheritableThreadLocal 与父子传递

普通 ThreadLocal 中，父线程 set 的值，子线程拿不到——因为每个线程有自己的 Map。`InheritableThreadLocal` 解决了这个问题：创建子线程时，JVM 会把父线程 ThreadLocalMap 中所有 InheritableThreadLocal 的 value 浅拷贝到子线程的 Map 中。

使用场景：全链路追踪中的 TraceId 传递。请求进入服务后生成一个 traceId 放入 InheritableThreadLocal，后续线程池中的子任务自动拿到同一个 traceId，日志串联不中断。

注意两点：一是 InheritableThreadLocal 的拷贝仅在**线程创建时**发生一次，之后父线程的修改对已创建的子线程不可见；二是线程池场景下线程是复用的，不会重新创建，InheritableThreadLocal 失效，需要改用阿里的 TransmittableThreadLocal 或手动在任务提交前传递上下文。

### 7.5 使用场景总结

- **线程封闭的单线程数据**：当前用户、当前请求、当前事务
- **避免参数层层传递**：链路追踪 ID、语言地区设置
- **线程安全的昂贵对象**：DateFormatter（SimpleDateFormat 线程不安全，每个线程持有一个实例）、Random 实例
- **框架内部实现**：Spring 的事务同步器、Hibernate 的 Session、MyBatis 的分页插件

## 8. 线程池（ThreadPoolExecutor）

### 7 大参数

```
corePoolSize   核心线程数          —— 常驻线程数，即使空闲也不回收（除非 allowCoreThreadTimeOut）
maxPoolSize    最大线程数           —— 能同时活跃的最大线程
keepAliveTime  非核心线程空闲存活时间  —— 超过此时长的空闲非核心线程被回收
unit           时间单位             —— keepAliveTime 的单位
workQueue      阻塞队列             —— 任务暂存处，核心线程满时新任务先入队
threadFactory  线程工厂             —— 自定义线程名、守护线程、优先级
handler        拒绝策略             —— 队列满且线程数=最大值时，新任务如何处理
```

调参的核心矛盾在于 `corePoolSize` + `workQueue` + `maxPoolSize` 三者的配合：任务先被核心线程消化，多余的先排队，队列撑不住才扩到最大线程。所以如果设了一个无界队列，`maxPoolSize` 永远不会被触发。

### 工作流程

```
提交任务
  ↓
核心线程数 < corePoolSize？ → 是 → 创建核心线程
  ↓ 否
队列未满？ → 是 → 入队等待
  ↓ 否
线程数 < maxPoolSize？ → 是 → 创建非核心线程
  ↓ 否
执行拒绝策略
```

### 4 种拒绝策略

- **AbortPolicy（默认）**：直接抛 `RejectedExecutionException`。任务不会丢，调用方能立刻感知过载，做降级或重试。最安全的选择。
- **CallerRunsPolicy**：由提交任务的线程（caller）自己执行这个任务。相当于把压力往回推，自然减速。缺点是 caller 被占用后无法继续提交新任务，且执行速度慢。
- **DiscardOldestPolicy**：丢弃队列里等待最久的任务，把新任务塞进去。适用于"新任务比旧任务更重要"的场景（如实时数据，最新的才有价值）。
- **DiscardPolicy**：静默丢弃新任务，不报错。非常危险，除非丢了不影响的场景（如非关键日志），否则不要用。

### 常用线程池的坑

| 线程池 | 风险 |
|--------|------|
| FixedThreadPool | LinkedBlockingQueue 无界 → OOM |
| CachedThreadPool | SynchronousQueue + maxPoolSize=Integer.MAX_VALUE → CPU 爆炸 |
| SingleThreadExecutor | LinkedBlockingQueue 无界 → OOM |

**为什么无界队列会导致 OOM**：`LinkedBlockingQueue` 默认容量是 `Integer.MAX_VALUE`，任务堆积速度超过处理速度时，队列不断膨胀直到撑爆堆内存。同理，`CachedThreadPool` 的 `SynchronousQueue` 不存任务、来一个就必须有一个线程接，而最大线程数又没上限，任务一涌进来就疯狂创建线程，CPU 全耗在线程创建和上下文切换上。

> **阿里巴巴规范**：强制用 ThreadPoolExecutor 手动创建，不允许用 Executors 工厂方法。因为 `Executors` 返回的线程池暗藏了上述无界队列和无界线程的坑，配置不可控。

## 9. 创建线程的方式

1. **继承 `Thread` 类**：重写 `run()`，然后 `start()`。最简单但扩展性差——Java 单继承，继承了 Thread 就不能再继承别的类，不适合复杂业务。
2. **实现 `Runnable` 接口**：实现 `run()`，传给 Thread 或线程池执行。不占用继承名额，任务和线程分离，推荐。
3. **实现 `Callable<T>` + `FutureTask`**：与 Runnable 唯一的区别是**有返回值**，且可以抛受检异常。通过 `Future.get()` 阻塞获取结果，`FutureTask` 既能当 Runnable 也能当 Future。
4. **线程池**：不直接 new Thread，而是通过 `ThreadPoolExecutor` 提交任务。核心优势是**线程复用**（避免反复创建/销毁的 CPU 和内存开销）和**资源管控**（控制并发上限）。

**推荐组合**：线程池 + Callable（需要返回值时）+ Runnable（不需要返回值时）。Thread 继承仅在学习阶段或极简 demo 中使用。

## 10. wait / notify vs LockSupport

- **`wait()` / `notify()`**：必须在 synchronized 块内调用，否则抛 `IllegalMonitorStateException`。wait 会释放锁并让出 CPU，被 notify 后不是立刻执行，而是先进入 BLOCKED 重新竞争锁，拿到锁后才继续执行。这是它和 sleep 的根本区别。

- **`LockSupport.park()` / `unpark()`**：不需要配合任何锁使用，随时可调用。最巧妙的设计是**许可证（permit）机制**——每个线程自带一个"许可证"，初始为 0。`unpark()` 先将许可证置为 1（重复调用不累加），`park()` 检查许可证：如果有就消费掉直接返回，如果没有才挂起线程。这意味着 **unpark 可以在 park 之前调用**，下一次 park 直接返回不阻塞。这个特性解决了 wait/notify 长期存在的"信号丢失"问题——notify 如果比 wait 先到达，wait 就永远等不到。

在 JUC 的底层实现中（AQS、FutureTask 等），内部用的全是 LockSupport，而不是 wait/notify。

## 11. JUC 常用工具类

- **CountDownLatch**：一组线程共同完成一件事，主线程等待所有人就绪。`countDown()` 减计数器，`await()` 阻塞直到计数器归零。**一次性的**，归零后不能重置。典型场景：主线程等多个数据源都加载完再开始处理。

- **CyclicBarrier**：N 个线程相互等待，全部到齐后集体出发。与 CountDownLatch 的关键区别：**可复用**（一个 barrier 可以跑多轮），且它是"相互等"（每个线程都在等别人），不是"一方等另一方"。还支持构造函数传入一个 `Runnable`，全部到齐后优先执行该任务，再集体释放。典型场景：多线程分片计算，每轮算完相互等待，合并结果后再进入下一轮。

- **Semaphore**：许可证池，控制同时访问资源的线程数。`acquire()` 拿一个许可（不够就等），`release()` 还一个许可。典型场景：限流（限制同时调下游接口的并发数）、连接池（限制同时持有的数据库连接数）。

- **Exchanger**：两个线程在汇合点交换数据。线程 A 到达后等待，线程 B 到达后双方同时交换数据并继续。典型场景：两个线程各自算一半数据，最后交换结果合并。

- **CompletableFuture**：异步任务编排工具，解决 Future 的痛点——Future 只能 `get()` 阻塞等结果，不能链式地"做完 A 再做 B，如果失败做 C"。`thenApply`（转换结果）、`thenCombine`（合并两个异步结果）、`exceptionally`（异常处理）、`allOf` / `anyOf`（并行等待全部/任意一个完成）等方法，可以无阻塞地编排复杂的异步调用链。典型场景：并行调多个下游接口，结果聚合，异常兜底。
