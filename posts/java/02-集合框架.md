# Java 集合框架

---

## 1. ArrayList vs LinkedList

| | ArrayList | LinkedList |
|--|-----------|------------|
| 底层结构 | `Object[]` 数组 | 双向链表 |
| 随机访问 | O(1) | O(n) |
| 头/尾增删 | O(n) / O(1) | O(1) |
| 内存占用 | 连续空间 | 节点额外存储 prev/next 指针 |
| 默认容量 | 10，扩容 1.5 倍 | — |

**扩容机制**：ArrayList 默认容量 10，每次扩容为原来的 1.5 倍（`oldCapacity + (oldCapacity >> 1)`），底层调用 `Arrays.copyOf` 复制整个数组。如果预估数据量，构造时给合理初始容量可以减少扩容开销。

**常见误区**："插入/删除 LinkedList 一定比 ArrayList 快"——不一定。头尾操作 LinkedList 确实 O(1)，但**中间位置插入**需要先遍历定位（O(n)），找到节点后才改指针（O(1)）。在数据量不大、CPU 缓存友好的情况下，ArrayList 的 `System.arraycopy` 连续拷贝反而可能比 LinkedList 的指针跳转更快。面试时要带前提说清。

**遍历选择**：ArrayList 用普通 for（下标直击），LinkedList 必须用迭代器或增强 for（否则每次 get(index) 都从头遍历，O(n²)）。判断依据是 `RandomAccess` 标记接口——ArrayList 实现了它，LinkedList 没有。

**选择建议**：99% 场景用 ArrayList。需要频繁头尾操作时 `ArrayDeque` 通常比 LinkedList 更好（也是数组实现，无链表指针开销，头尾 O(1)）。LinkedList 还实现 Deque 接口，需要双端队列 + List 双重能力时可以考虑。

## 2. HashMap 原理（超高频）

### 2.1 数据结构演变

- **JDK 1.7**：数组 + 链表。头插法——新节点插在链表头部，扩容 transfer 时也是头插。**多线程并发扩容时可能形成循环链表，导致 CPU 100%（下一节详述）**。
- **JDK 1.8**：数组 + 链表/红黑树。尾插法——新节点插在链表尾部，杜绝了循环链表。链表长度 ≥ 8 且数组长度 ≥ 64 时转为红黑树；树节点数 ≤ 6 时退化回链表。

为什么树化阈值是 8？源码注释给出了 Poisson 分布计算：在负载因子 0.75 下，一个桶中链表长度达到 8 的概率约为千万分之六，属于极小概率事件。所以 8 是"几乎不可能自然发生，一旦发生说明哈希冲突严重，值得树化"的分界点。退化阈值设为 6 而非 8 是为了留缓冲区间，避免频繁转换。

### 2.2 hash 扰动函数

HashMap 不直接使用 key 的 `hashCode()`，而是做一次扰动：

```java
// JDK 1.8
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

**为什么要扰动？** `(n - 1) & hash` 取模时，当容量较小（比如 16，n-1 = 15 = 0b1111），只用到了 hash 的低 4 位。高 16 位完全被忽略——如果两个对象的低几位碰巧相同，即使 hash 值完全不同，也会打进同一个桶。扰动把高 16 位和低 16 位做异或，**让高位特征混入低位**，减少低位哈希冲突。

JDK 1.7 扰动更复杂（4 次移位 + 异或），JDK 1.8 简化为一次——因为引入了红黑树，即使在极端冲突下查找也从 O(n) 降到 O(log n)，扰动没必要做那么重。

### 2.3 put 流程详解

```
1. 计算 hash = (h = key.hashCode()) ^ (h >>> 16)
2. 如果 table 为空 → resize() 初始化（延迟初始化，节省内存）
3. i = (n - 1) & hash  → 定位桶下标
4. 桶为空 → 直接 new Node 放入
5. 桶非空 → 判断首个节点：
   a. 首节点 hash 和 key 都相同 → 覆盖
   b. 首节点是 TreeNode → 走红黑树 putTreeVal()
   c. 否则 → 遍历链表：
      - 找到相同 key → 跳出，覆盖
      - 遍历到尾 → 尾插新节点，判断是否 ≥ 8 → treeifyBin()
6. 覆盖或插入后 modCount++
7. size > threshold（capacity × loadFactor）→ resize()
```

两个细节：
- **延迟初始化**：`new HashMap<>()` 不会立即分配数组，第一次 put 时才 `resize()` 初始化。这样空 Map 不占内存。
- **treeifyBin 的双重检查**：链表长度 ≥ 8 后，先检查数组长度是否 ≥ 64——如果不到 64，认为数组太挤，优先扩容而非树化（扩容也能缩短链表）。

### 2.4 扩容机制

- 默认初始容量 **16**，负载因子 **0.75**。
- 扩容为原来的 **2 倍**（`newCap = oldCap << 1`）。
- **为什么容量必须是 2 的幂？**
  - `(n - 1) & hash` 等价于 `hash % n`，位运算远比取模快。
  - 扩容后节点重分布时，只需判断 `hash & oldCap`：为 0 则位置不变，为 1 则移到**原位置 + oldCap**。因为 n 是 2 的幂，扩容前后 n-1 的二进制只差一个最高位的 1。例如 16→32：旧掩码 01111，新掩码 11111，多出的那一位决定了位置是否偏移。
- **负载因子的含义**：0.75 是时间与空间的权衡。太高 → 数组拥挤，冲突增多，查找变慢；太低 → 频繁扩容，空间浪费。0.75 是 HashMap 作者通过统计实验得出的经验值。

扩容时 JDK 1.8 的核心优化：不再像 1.7 那样逐个 rehash 插入，而是把同一条链上的节点拆成两条链（不需要移动的和需要移动的），各自整体迁移，效率更高。

### 2.5 树化与退树化

**树化条件**（两个同时满足）：
1. 单桶链表长度 ≥ 8（`TREEIFY_THRESHOLD`）
2. 数组长度 ≥ 64（`MIN_TREEIFY_CAPACITY`）——否则优先扩容

**退树化条件**：
- resize 时或 remove 后，树节点数 ≤ 6（`UNTREEIFY_THRESHOLD`）→ 退化为链表
- 注意：退化阈值 6 ＜ 树化阈值 8，中间差值为 1，防止大小在阈值附近波动导致反复转换

红黑树存在的意义不是常规场景下的性能提升（常规场景下链表够短），而是**防止恶意攻击**——如果攻击者精心构造一批哈希碰撞的 key，链表退化到 O(n)，HashMap 直接变成链表。红黑树保证了无论如何冲突，查找都是 O(log n)。

### 2.6 JDK 1.7 头插法死循环

并发扩容时可能形成循环链表，get 时陷入死循环 CPU 100%。过程：

1. 线程 A 和线程 B 同时触发扩容，各自创建新数组
2. 1.7 的 `transfer()` 从头遍历旧链表，**头插**到新数组。假设旧链表 A→B→null
3. 线程 B 先完成 transfer，新链表变成 B→A→null
4. 线程 A 苏醒，它的局部变量 `e=A, next=B`，但此时 B.next 已被线程 B 改成指向 A
5. 线程 A 继续头插：A 插入新桶 → 处理 B 时 B.next=A → 插入 B 后，A 还在链上，形成 B⇄A 循环

JDK 1.8 改尾插法从根本上杜绝了这个问题——遍历顺序与插入顺序一致，不会颠倒链表方向。

### 2.7 线程安全问题

HashMap 非线程安全，多线程下的典型问题：

- **put 数据丢失**：两个线程同时 put 不同 key 到同一个空桶，各自判断为空后插入，后一个覆盖前一个。
- **size 不准确**：`size++` 非原子操作，多线程 put 后统计值可能小于实际元素数。
- **JDK 1.7 扩容死循环**（见上节）。
- **JDK 1.8 扩容数据丢失**：`transfer` 中链表拆链操作非原子，可能丢节点。

替代方案：`ConcurrentHashMap`（推荐）、`Collections.synchronizedMap()`（不推荐，锁全表，性能差）。

### 2.8 HashMap 与 Hashtable

| | HashMap | Hashtable |
|--|---------|-----------|
| 线程安全 | ❌ | ✅ synchronized 锁全表 |
| null | 允许 key/value 为 null | ❌ 不允许 |
| 效率 | 高 | 低（几乎淘汰） |
| 扩容 | 2 倍 | 2n + 1 |
| 默认容量 | 16 | 11 |
| 继承 | AbstractMap | Dictionary |

Hashtable 已被淘汰。单线程用 HashMap，多线程用 ConcurrentHashMap。

## 3. ConcurrentHashMap

### 3.1 JDK 1.7 分段锁

1.7 的 CHM 把整个数组拆成多个 Segment，每个 Segment 内部是一个小 HashMap（HashEntry 数组 + 链表），Segment 继承 `ReentrantLock`。

**写入流程**：先根据 key 的高位 hash 定位 Segment → 对该 Segment 加锁 → 在 Segment 内部再 hash 定位 HashEntry → 插入/覆盖。读取不需要锁（HashEntry 的 value 是 volatile 保证可见性）。

**默认并发度 16**：即 16 个 Segment，意味着最多 16 个线程同时写。构造后并发度不可变——扩容是在 Segment 内部扩，Segment 数量始终不变。

**局限性**：
- 跨 Segment 操作（如 `size()`）需要锁住所有 Segment，代价高
- 并发度固定，不可动态扩展
- 每个 Segment 的锁相互独立，操作同一 Segment 的线程仍会阻塞

### 3.2 JDK 1.8 设计思想

1.8 彻底废弃分段锁，回归单层数组结构，每个桶独立竞争：

- **读**：完全无锁。Node 的 `val` 和 `next` 都是 volatile，保证可见性。
- **写（桶为空）**：CAS 自旋尝试设置首节点（`tabAt` + `casTabAt`）。
- **写（桶非空）**：synchronized 锁住桶的**头节点**（而非整个 Segment），粒度细到单个桶。
- **红黑树**：树节点加写锁的方式同链表——锁头节点。

**为什么从 ReentrantLock 改 synchronized？**
- JDK 1.6 后 synchronized 经历了锁升级（偏向锁→轻量级锁→重量级锁），在低竞争场景下性能大幅提升，不再比 ReentrantLock 差
- synchronized 的锁对象直接是桶头节点，语义自然且内存更紧凑
- ReentrantLock 每个 Segment 是一个独立对象，额外内存开销

### 3.3 put 流程详解

```java
// 核心流程（对应源码 putVal）
final V putVal(K key, V value, boolean onlyIfAbsent) {
    // 1. key/value 不能为 null（与 HashMap 不同）
    int hash = spread(key.hashCode());  // 扰动：HASH_BITS 屏蔽符号位
    for (Node<K,V>[] tab = table;;) {  // 自旋
        // 2. table 为空 → CAS 初始化（多线程共享初始化）
        // 3. 桶为空 → CAS 插入首节点，成功则跳出
        // 4. 桶首节点 hash == MOVED → 正在扩容，当前线程帮忙迁移（helpTransfer）
        // 5. 否则 → synchronized 锁住首节点，遍历链表/树：
        //    - 找到相同 key → 覆盖（onlyIfAbsent 控制）
        //    - 未找到 → 尾插，遍历中 binCount 计数
        // 6. 链表长 ≥ 8 → treeifyBin（内部双重检查，同 HashMap）
    }
    addCount(1L, binCount);  // 计数 + 检查是否需要扩容
}
```

关键区别：
- **key/value 不能为 null**：与 HashMap 不同，CHM get 的返回值可能为 null（表示不存在），如果允许存 null 则无法区分"值为 null"和"key 不存在"
- **自旋 + CAS**：外层 for 循环等效 spin，失败重试而非阻塞等待
- **helpTransfer**：检测到 ForwardingNode 时主动协助扩容，利用多线程加速

### 3.4 多线程并发扩容

CHM 1.8 的扩容支持多线程协作，核心机制：

- **sizeCtl**：多义字段——正数 = 扩容阈值，负数 = 正在扩容（低 16 位记录参与线程数 + 1）
- **transferIndex**：倒序步进指针，标记"下一个待迁移桶的索引"。每个线程领取一个 stride 区间（最少 16 个桶），倒序处理
- **ForwardingNode**：桶迁移完成后放置的特殊节点，hash=MOVED。get 遇到它会去新数组查找；put 遇到它会协助迁移

流程：
1. 触发阈值 → 线程 A 启动扩容，创建新数组（2 倍大小），设置 `sizeCtl = -2`（1 + 1 个线程）
2. 线程 A 从 `transferIndex` 领取一段桶区间进行迁移
3. 线程 B 写入时遇到 ForwardingNode → 检测到正在扩容 → 修改 `sizeCtl`（线程数 + 1）→ 也领取一段 bucket 迁移
4. 所有节点迁移完成后，用旧数组的 ForwardingNode 指向新数组；sizeCtl 重置为新的扩容阈值

### 3.5 size 计数：CounterCell 累加

HashMap 的 `size` 是一个 int，但 CHM 多线程并发 put 时不可能给 size 加锁。1.8 采用类似 LongAdder 的分段累加方案：

- **baseCount**：基础计数，低竞争时 CAS 直接修改
- **CounterCell[]**：CAS 竞争失败时，线程在自己的 CounterCell（随机或 hash 分配）里累加，最后 `size = baseCount + 所有 CounterCell 的 value`
- **addCount()**：每次 put/remove 后调用，先 CAS 更新 baseCount，失败则路由到 CounterCell

这种方式将热点分散到多个 CounterCell，避免 JDK 1.7 中 `size()` 锁所有 Segment 的高昂代价。

### 3.6 get 为什么不用加锁

- 数组通过 `Unsafe.getObjectVolatile()` 读取，保证拿到最新的引用
- Node 的 `val` 和 `next` 都是 volatile，可见性由 JMM 保证
- 即使正在扩容，get 读到 ForwardingNode 后会去新数组继续查找，不会读到脏数据

### 3.7 1.7 vs 1.8 对比

| | JDK 1.7 | JDK 1.8 |
|--|---------|---------|
| 结构 | Segment[] + HashEntry[] | Node[] + 链表/红黑树 |
| 锁 | 分段锁（Segment 继承 ReentrantLock） | CAS + synchronized 锁头节点 |
| 锁粒度 | 16 个 Segment（不可变） | 桶级别（随扩容变细） |
| 扩容 | Segment 内部独立扩 | 多线程协作扩 |
| size | 锁所有 Segment，不锁可能不准 | CounterCell 累加 |
| 读 | 无锁，value volatile | 无锁，volatile + ForwardingNode |
| 并发度 | 固定 | 动态，理论无上限 |
| null | key/value 都不允许 | key/value 都不允许 |

### 3.8 与其他并发 Map 对比

| | ConcurrentHashMap | Hashtable | Collections.synchronizedMap |
|--|-------------------|-----------|----------------------------|
| 锁 | CAS + synchronized 桶级 | synchronized 锁方法（全表） | synchronized 锁 mutex 对象 |
| 读并发 | 读完全无锁 | 读也加锁 | 读也加锁 |
| 扩容 | 多线程协作 | 单线程，锁全表 | 单线程，锁全表 |
| null | ❌ | ❌ | 看被包装的 Map |

结论：任何并发场景直接用 ConcurrentHashMap，Hashtable 和 synchronizedMap 已淘汰。

## 4. HashSet / TreeSet / LinkedHashSet

- **HashSet**：基于 HashMap，元素作为 key，value 固定为 `new Object()`（一个哑元占位符，称为 PRESENT）。利用了 HashMap 的 key 唯一性保证集合元素不重复。add/remove/contains 均是 O(1)。
- **TreeSet**：基于 TreeMap，红黑树实现。元素有序（自然顺序或传入 Comparator），增删查 O(log n)。要求元素实现 Comparable 或构造时传 Comparator。`first()`/`last()`/`subSet()` 等导航方法来自 SortedSet 接口。
- **LinkedHashSet**：继承 HashSet，底层是 LinkedHashMap，在 HashMap 的基础上用双向链表维护**插入顺序**。遍历顺序与插入顺序一致，但额外内存开销比 HashSet 高（每个节点存 prev/next 指针）。

三者的共同点：**都不可重复**，都允许 null（TreeSet 排序时需要比较 null 会抛 NPE，不建议）。线程不安全，并发场景用 `Collections.synchronizedSet()` 或 `ConcurrentSkipListSet`。

## 5. TreeMap

- **底层**：红黑树，一种自平衡二叉查找树，每个节点有红/黑颜色属性，通过旋转和变色维持平衡（最长路径不超过最短路径的 2 倍），保证增删查都是 O(log n)。
- **排序规则**：key 必须可比较——要么实现 `Comparable`（自然顺序），要么构造时传 `Comparator`（自定义顺序）。两者都提供时优先用 Comparator。
- **key 不能为 null**：因为需要比较排序，null 调用 compareTo 会抛 NPE。value 可以为 null。
- **核心方法**：`firstKey()`/`lastKey()` 取首尾 key、`subMap(from, to)`/`tailMap(from)` 取子范围、`ceilingKey(k)` 取 ≥ k 的最小 key、`floorKey(k)` 取 ≤ k 的最大 key。这些导航方法源自 `NavigableMap` 接口。
- **与 HashMap 对比**：HashMap 无序 O(1)，TreeMap 有序 O(log n)。需要排序或范围查询时选 TreeMap，其他场景 HashMap。
- **替代**：并发场景用 `ConcurrentSkipListMap`（跳表实现，非阻塞，同样有序），TreeMap 本身线程不安全。

## 6. fail-fast 与 fail-safe

| | fail-fast | fail-safe |
|--|-----------|-----------|
| 机制 | 迭代过程中结构修改 → 抛 ConcurrentModificationException | 迭代副本 |
| 检测方式 | modCount != expectedModCount | — |
| 代表 | ArrayList, HashMap | CopyOnWriteArrayList, ConcurrentHashMap |
| 优缺点 | 快速失败 | 数据可能过时但不会炸 |

## 7. Collections 工具类

- `Collections.synchronizedList()`：返回线程安全的包装 List（锁所有方法）
- `Collections.unmodifiableList()`：返回不可变 List
- `Collections.sort()`：底层 Arrays.sort()，元素量小时用快排，大时用 TimSort
