## 2. TCP 三次握手 / 四次挥手

### 三次握手

```
Client                            Server
  |─── SYN (seq=x) ─────────────→ |
  |←── SYN+ACK (seq=y, ack=x+1) ─ |
  |─── ACK (ack=y+1) ───────────→ |
  |             ESTABLISHED       |
```

**为什么三次而不是两次？** 防止失效的连接请求到达服务器，服务器误以为建立了连接。

**为什么不是四次？** 握手没有"半关闭"，可以合并服务端的 SYN 和 ACK。

### 四次挥手

```
Client                            Server
  |─── FIN (seq=u) ─────────────→ |
  |←── ACK (ack=u+1) ──────────── |  ← 半关闭：Server 可能还有数据要发
  |←── FIN (seq=v, ack=u+1) ──── |
  |─── ACK (ack=v+1) ───────────→ |
  |   TIME_WAIT (2MSL)            |
```

**为什么四次？** TCP 是全双工，一方关闭发送但另一方可能还在传数据（半关闭状态）。

**TIME_WAIT 为什么等 2MSL？**
- 确保最后的 ACK 能到达（对方超时会重发 FIN）
- 让本连接的报文在网络中彻底消失，不影响新连接

## 3. TCP 可靠性保证

- **序号 + 确认应答**：每个字节编号，收到 ACK
- **超时重传**：RTT 动态计算超时时间，超时重发
- **滑动窗口**（流量控制）：接收方告知可接收窗口大小，发送方控制速度
- **拥塞控制**：慢启动 → 拥塞避免 → 快重传（3 个重复 ACK）→ 快恢复

---

## 4. TCP 拥塞控制

```
慢启动：cwnd 从 1 开始指数增长（每 RTT 翻倍）
  ↓ 达到 ssthresh 阈值
拥塞避免：cwnd 线性增长（每 RTT +1）
  ↓ 超时
慢启动：ssthresh = cwnd/2，cwnd = 1
  ↓ 收到 3 个重复 ACK
快重传 + 快恢复：ssthresh = cwnd/2，cwnd = ssthresh，进入拥塞避免
```

## 5. HTTP 各版本

### HTTP/1.0

- 默认短连接（`Connection: keep-alive` 才长连）
- 队头阻塞（一个请求阻塞后续）

### HTTP/1.1

- 默认长连接
- 管线化（Pipelining，仍队头阻塞）
- Host 头（一台服务器多个虚拟主机）
- 断点续传（Range 头）
- 缓存策略增强（Cache-Control）

### HTTP/2

- **多路复用**：帧 + 流，一个 TCP 连接承载多个请求（无队头阻塞）
- **头部压缩**：HPACK 算法，静态表 + 动态表
- **服务器推送**（Server Push）
- 二进制分帧、流量控制、流优先级
- 问题：TCP 层面仍有队头阻塞（丢包需整窗口重传）

### HTTP/3

- 基于 **QUIC**（UDP）
- 0-RTT 握手（缓存过的连接可立即发数据）
- 彻底解决 TCP 队头阻塞
- 连接迁移（WiFi → 4G 无缝切换）

## 6. HTTPS 握手

```
1. ClientHello：支持的 TLS 版本 + 加密套件 + 客户端随机数
2. ServerHello：选定的加密套件 + 服务器随机数 + 证书
3. 客户端验证证书（CA 签名链）
4. 生成 PreMaster Secret → 服务器公钥加密 → 发给服务器
5. 双方用三个随机数（客户端随机 + 服务端随机 + PreMaster）→ 计算会话密钥
6. 后续对称加密通信
```

## 7. DNS 解析流程

```
浏览器缓存 → 操作系统 hosts 文件 → 本地 DNS 服务器
  → 根 DNS（.）→ 顶级域 DNS（.com）→ 权威 DNS（example.com）
  → 返回 IP
```

- DNS 默认用 UDP（快，包小），区域传输用 TCP

## 8. 从输入 URL 到页面展示

```
1. URL 解析
2. DNS 解析（域名 → IP）
3. TCP 三次握手
4. TLS 握手（HTTPS）
5. 发送 HTTP 请求
6. 服务器处理并返回响应
7. 浏览器解析 HTML → 构建 DOM 树
8. 解析 CSS → 构建 CSSOM 树
9. 合并 → Render Tree
10. 布局（Layout / Reflow）
11. 绘制（Paint）
12. 合成（Composite）
```

## 9. GET vs POST

| | GET | POST |
|--|-----|------|
| 幂等性 | ✅ | ❌ |
| 参数位置 | URL 查询字符串 | 请求体 |
| 缓存 | 可缓存 | 默认不可缓存 |
| 安全性 | 参数暴露在 URL | 相对隐蔽（仍可抓包） |
| 长度限制 | 浏览器限制（约 2KB） | 无限制 |
| 语义 | 获取资源 | 创建/提交 |

> 本质区别：语义不同。底层都是 TCP，GET 也可带 body（不推荐）

## 10. Cookie / Session / Token

| | Cookie | Session | Token (JWT) |
|--|--------|---------|-------------|
| 存储位置 | 浏览器 | 服务器 | 客户端 |
| 安全性 | 可被篡改、XSS | Cookie 劫持 | 可签名防篡改 |
| 扩展性 | — | 服务器存 Session → 分布式需共享 | 无状态，天然支持分布式 |
| 缺点 | 大小限制 4KB | 服务器内存、共享问题 | 无法主动失效 |

## 11. TCP 与 UDP

| | TCP | UDP |
|--|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠（确认、重传） | 不可靠 |
| 顺序 | 有序 | 无序 |
| 速度 | 慢 | 快 |
| 场景 | HTTP、文件传输、邮件 | DNS、视频直播、在线游戏 |
| 头部 | 20~60 字节 | 固定 8 字节 |
